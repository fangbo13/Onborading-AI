===== SLIDE 4 =====
关键问题聚焦 1/5
问题 1：个人设置入口重复
侧边栏导航菜单 & 顶部用户头像下拉均有「个人设置」入口
[截图占位]
用户打开侧边栏看到 Profile 菜单项，又在右上角头像下拉中看到同样的 Profile 选项，产生困惑：这两个是同一个功能吗？
影响分析（认知心理学视角）
Hick 定律：选择越多，决策时间越长。重复入口增加认知负担
一致性原则违反：用户预期每个功能只有一个明确入口
新用户会疑惑「我应该从哪里进入设置？」产生犹豫
资深用户可能从不同入口进入后，发现页面相同，觉得产品不精致
维护成本高：如果将来设置页面拆分，两个入口的语义可能不同步
改进方案
方案 A（推荐）：移除侧边栏 Profile 菜单项，仅保留顶部用户头像下拉中的设置入口
方案 B：保留侧边栏 Profile 入口，移除顶部下拉中的设置，仅保留 Logout
方案 C：侧边栏 Profile 入口点击后改为弹出设置面板（Drawer），区分交互深度

===== SLIDE 5 =====
关键问题聚焦 2/5
问题 2A：侧边栏当前仅为导航菜单，缺少对话列表
产品规划采用 DeepSeek 样式，但当前实现尚未包含对话列表功能
[截图占位]
当前侧边栏只有 4 个导航项（对话/历史/知识库/个人设置），用户需要点击「历史」才能查看过去的对话，与 DeepSeek 的常驻对话列表不同。
影响分析（认知心理学视角）
DeepSeek 模式的核心价值：对话历史在视线范围内，一键切换
当前设计将「对话」和「历史」割裂为两个页面，用户需离开对话页面才能查看历史
用户心理模型：对话 = 连续的工作流。割裂为两个页面打破了这个模型
效率损失：每次切换对话至少需要 3 步操作（点历史 -> 找对话 -> 点返回）
与产品规划目标（DeepSeek 样式侧边栏）存在实现差距
改进方案
方案 A（推荐）：在侧边栏中新增对话列表区域，位于导航菜单下方，按时间分组
方案 B：将 HistoryPage 的日期分组逻辑复用为 Zustand selector
方案 C：折叠态仅显示最近3条对话的圆点指示器，展开态显示完整列表
注意：侧边栏宽度（240px）需要容纳列表，标题截断需加 tooltip

===== SLIDE 6 =====
关键问题聚焦 3/5
问题 2B：DeepSeek 风格侧边栏的体验风险分析
空间争夺、注意力干扰、信息承载极简、新手空状态、管理操作、定位效率、切换同步
[截图占位]
左侧固定宽度侧边栏与右侧对话区的视觉竞争关系。单行标题截断、时间分组、长按操作在窄栏中的可用性挑战。
影响分析（认知心理学视角）
空间争夺：240px 侧边栏在 1366px 屏幕上占 17.6%；在 1024px 平板上占 23.4%，明显挤压
注意力干扰：时间分组标题（今天/昨天）增加了视觉噪音，视线左右切换产生认知疲劳
信息承载极简：单行标题截断后无法区分相似对话。缺失日期、轮次、最后消息预览
新手空状态：无对话时侧边栏列表区域应显示引导提示，而非空白
管理操作：240px 宽度内放置重命名/删除按钮空间不足，需要右键菜单或长按触发
定位效率：侧边栏列表无全文搜索时，20+ 对话查找效率低于独立 HistoryPage
改进方案
增加可拖拽调整侧边栏宽度（180px-350px），用户自适应
非当前对话的列表项降低透明度至 60%，减少视觉竞争
hover 时显示 tooltip（完整标题 + 日期 + 轮次 + 首句预览）
0 对话时显示「开始你的第一次对话」引导卡片
右键菜单（桌面端）+ 长按 500ms（移动端），避免按钮拥挤
侧边栏顶部集成搜索胶囊，300ms debounce，实时过滤

===== SLIDE 7 =====
关键问题聚焦 4/5
问题 2C：对话列表信息承载极简的深层体验问题
单行标题截断、标题重复/为空、缺少辅助信息的用户体验隐患
[截图占位]
当标题过长被截断（...）、多个对话标题相似或重复、智能标题生成失败时（如输入「你好」），用户无法区分不同对话。
影响分析（认知心理学视角）
识别性危机：3 个关于「年假政策」的对话在列表中都显示为相同截断文本
空标题：用户输入无意义内容时，智能标题回退为「新对话」，多个条目并存
时间分组局限：按时间分组解决了粗略定位，但无法解决同一天内相似对话区分
认知负荷：用户需要逐个点击进入才能确认哪个是自己要找的对话
Fitts 定律：侧边栏列表项高度较小时，点击精度降低，尤其在触控场景下
改进方案
标题增强：列表项显示标题 + 副标题（日期 + 轮次），如「年假政策 - 昨天 · 12轮」
颜色编码：按对话主题自动着色小圆点，辅助视觉区分
智能去重：相同标题的对话追加序号后缀
Hover 预览：鼠标悬停时展开为卡片式预览
搜索增强：搜索不仅匹配标题，还匹配对话内容（全文搜索）

===== SLIDE 8 =====
关键问题聚焦 5/5
问题 3：无全局 Error Boundary + 边界状态处理缺失
React 应用缺少错误边界组件，无断网检测、无离线提示
[截图占位]
任何未捕获的 React 错误将导致整页白屏。断网后用户无感知地发送消息，等待超时后看到生硬错误。
影响分析（认知心理学视角）
单点故障：一个组件的渲染错误（如 Markdown 解析异常）可导致整个应用白屏
信任崩塌：白屏是用户最恐惧的体验之一，直接传达「产品不可靠」的信号
断网无感知：用户输入长消息后点击发送，因断网而长时间等待
恢复成本：白屏后用户需要手动刷新页面，之前的输入和对话状态全部丢失
企业环境风险：企业内网波动频繁，断网是高频场景，不是边缘情况
改进方案
添加全局 ErrorBoundary 组件，包裹 <Outlet />，错误时显示友好降级 UI + 重试按钮
添加网络状态监听：检查 navigator.onLine，监听 online/offline 事件，断网时顶部显示红色横幅
消息发送前检测网络状态，断网时禁用发送按钮并提示
SSE 流式连接超时处理：超过 30 秒无 token 到达时自动断开并提示重试

===== SLIDE 9 =====
其他体验问题（P0 严重 + P1 中等）
严重
#5  对话标题截断无完整展示
标题超过 30 字符被截断后，用户无法看到完整标题
为什么不好：标题是用户识别对话的首要线索。截断后丢失关键信息，违反「状态可见性」原则。
怎么改更好：CSS text-overflow: ellipsis + Antd Tooltip，hover 时显示完整标题。
严重
#6  移动端消息操作无视觉反馈
移动端通过长按 500ms 触发操作菜单，但长按期间无视觉反馈
为什么不好：操作无反馈违反 Nielsen 启发式评估第1条「系统状态可见性」。用户会重复长按或放弃操作。
怎么改更好：长按时目标消息增加高亮边框 + 缩放动画 + 触觉反馈(navigator.vibrate)。
中等
#7  知识管理页面硬编码英文
KnowledgeBasePage 上传验证提示使用英文硬编码，未使用 t()
为什么不好：与其他页面完善的 i18n 不一致，中文用户看到英文错误提示会产生困惑和不信任感。
怎么改更好：在 admin.json 中添加翻译，替换硬编码字符串为 t() 调用。
中等
#8  消息加载仅 Spinner，无骨架屏
加载历史消息时显示居中的 Spinner + Loading，无内容预览
为什么不好：骨架屏通过占位符降低认知不确定性，比纯 Spinner感知等待时间短 30%。
怎么改更好：添加 Skeleton 组件：3-5 条模拟消息气泡，使用 shimmer 动画。
中等
#9  切换对话无 loading 过渡
HistoryPage 点击对话后，消息区域瞬间替换内容，无过渡动画
为什么不好：突然的内容切换违反用户预期，大脑需要额外认知资源处理视觉变化，产生轻微眩晕感。
怎么改更好：切换时添加 200ms 淡入淡出过渡，或显示短暂 skeleton 过渡态。
中等
#10  侧边栏内无对话搜索
切换到 DeepSeek 式侧边栏后，对话数量增多时查找效率急剧下降
为什么不好：滚动查找违反 Hick 定律和 Fitts 定律。用户在长列表中滚动是最低效的导航方式。
怎么改更好：侧边栏顶部集成搜索输入框（胶囊样式），复用 useDebounce 300ms，实时过滤标题匹配项。

===== SLIDE 10 =====
其他体验问题续（P1 中等 + P2 建议）
中等
#11  响应式断点不统一
AppLayout 使用 768px，LoginPage 使用 800px，768-800px 区间行为不一致
为什么不好：断点不一致导致特定窗口宽度下布局异常或交互混乱，用户调整窗口大小时体验不可预测。
怎么改更好：提取统一的 useBreakpoint hook，定义 breakpoints 对象全局复用。
中等
#12  aria-live 文本裁切不当
流式输出的 aria-live 使用 slice(-100) 裁切，可能从单词中间截断，读屏体验差
为什么不好：读屏用户听到的是不完整的单词或句子片段，理解困难。违反 WCAG 4.1.3 状态消息规范。
怎么改更好：改为按句子边界裁切（找最后一个句号/换行），或仅播报完整的最新消息块。
建议
#13  WelcomeScreen 卡片 hover 非 CSS 过渡
快捷提问卡片的 hover 效果通过 inline JS 直接修改 style，无 transition 属性，动画生硬
为什么不好：没有 transition 的样式变化是瞬时的，缺少中间态导致视觉跳跃感，降低产品精致感。
怎么改更好：改用 CSS 类 + transition: all 0.2s ease，在 globals.css 中定义hover 类，组件只负责切换 className。
建议
#14  Skip-to-content 使用内联 JS
无障碍跳过链接通过 onFocus/onBlur 内联修改 style.top，而非 CSS 类切换
为什么不好：内联样式与 CSS 变量体系脱节，在深色模式下可能出现对比度不足的问题。
怎么改更好：定义 .skip-link CSS 类，使用 :focus-visible 伪类控制显示，与全局 focus-visible 样式体系一致。
建议
#15  HistoryPage 无分页性能风险
HistoryPage 一次性加载所有会话，无分页或虚拟滚动，对话量大时渲染和搜索性能下降
为什么不好：大量 DOM 节点同时存在会降低渲染性能，用户在长列表中滚动可能出现卡顿。
怎么改更好：实现虚拟滚动（react-window）或分页加载（每页 20 条），搜索时仅渲染匹配结果。
建议
#16  流式滚动用 instant 非 smooth
ChatPage 流式期间使用 scrollIntoView({behavior:'instant'})而非 'smooth'，消息区域频繁跳动
为什么不好：instant 滚动是瞬时跳变，用户视线跟不上新内容生成速度，产生视觉不适。
怎么改更好：实现智能节流：每 500ms 最多滚动一次，或使用scroll-behavior: smooth 的全局 CSS 作为回退。

Hick 定律（选择复杂度）
选择数量与决策时间成对数关系
|  侧边栏窄列表项（36px）在 20+ 条目时，底部条目点击精度低  |  移动端长按目标区域小，手指遮挡导致操作困难
改进: 列表项高度增至 44px（触控友好），长按操作目标增大，复制按钮常驻或增大热区
Fitts 定律（目标可达性）
点击时间与目标距离和大小成比例
|  对话列表按 5 个时间组展示（今天/昨天/7天/30天/更早），接近上限  |  Onboarding Tour 4 步引导 + 初始 Modal 共 5 步，信息量饱和
改进: 时间组可折叠（默认只展开今天和昨天），引导分两轮进行
Miller 定律（认知容量）
工作记忆容量约 7+-2 个信息块
|  侧边栏导航菜单和对话列表需要明确的视觉分隔  |  消息气泡的 hover 操作按钮与气泡内容视觉上割裂
改进: 侧边栏用分割线+标签区分导航区和对话区，Header 用间距或分割线分组
格式塔原理（视觉组织）
用户倾向于将视觉元素组织为有意义的整体
|  流式输出的打字机效果是正面峰值，但白屏错误是极强的负面峰值  |  退出登录无确认直接跳转（终值），用户可能误触退出
改进: 加强正面峰值（加载骨架屏+平滑过渡），消除负面峰值（ErrorBoundary+退出确认）
峰值-终值定律（体验记忆）
用户记忆主要由体验的峰值和终值决定
